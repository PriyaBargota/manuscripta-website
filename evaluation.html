Critical Evaluation

User Interface and User Experience

The teacher application received consistently positive feedback from all five teaching professionals who tested it. The hierarchical content library, cascading deployment dropdowns and colour-coded device grid were described as intuitive by testers with standard computer literacy. The clean separation between the Library, Classroom and Responses views reduces cognitive load for teachers managing a live lesson.

The Android student interface was deliberately minimal — large monochromatic text, a single persistent Raise Hand button, and no navigation chrome beyond what is necessary for the current task. This directly addresses the design principle established in C1 and aligns with what the Richard Cloudesley School teachers described as a key requirement. The interface has not yet been formally tested with SEN students, which remains a significant gap.

Functionality

All thirteen Must Have functional requirements were delivered. Three of the six Should Have requirements were delivered in full, with the remaining three partially implemented. The AI content generation pipeline represents the strongest functional achievement — generating formatted worksheets and quizzes from source documents with no internet connection, running entirely on the teacher's laptop.

The teacher oversight model for AI feedback was a deliberate design decision validated by stakeholder feedback. Rather than sending AI-generated marks directly to students, the teacher reviews and approves each piece of feedback before it is delivered. This addresses the concern raised at the showcase that auto-marking without teacher control could undermine trust in the system.

AI Output Quality

Manuscripta's AI pipeline introduces non-determinism — the same prompt does not always produce identical output. This is an inherent property of large language models and cannot be eliminated entirely. The system addresses this through a deterministic post-processing pipeline that normalises output regardless of model variance: unclosed code blocks are closed, headers above H3 are normalised, malformed question markers are fixed and invalid references are removed.

⚠ Systematic AI output quality testing — running repeated prompts to measure output variance and verify factual accuracy — was not completed within the project timeline. Manual testing confirmed that the post-processing pipeline consistently produces structurally valid output. Factual accuracy of AI-generated content depends on the quality of source documents provided by the teacher and the accuracy of the underlying Qwen3:8b model. Teachers are advised to review all generated content before deployment, which is enforced by the system design — materials cannot be distributed without passing through the teacher's content library first.

Stability and Reliability

The system operates reliably over a local area network for single-session classroom use. The 3-second heartbeat mechanism provides real-time device status updates and detects disconnections within 10 seconds. The Android Room database ensures student work is never lost during network interruptions, addressing the concern raised by Dean Mohamedally that session data must persist locally.

⚠ Long-session stability and multi-session reliability have not been formally tested. The system was validated for single lesson scenarios of approximately 60 minutes. Behaviour over extended deployment periods or across multiple consecutive sessions is not yet characterised.

Performance

⚠ Formal performance benchmarks were not completed within the project timeline. The non-functional requirements specified content distribution to 30 devices within 10 seconds and dashboard updates within 3 seconds. Manual testing with a small number of connected devices suggests these targets are achievable on a standard school Wi-Fi network, but this has not been validated at the full 30-device scale.

AI generation time varies significantly by hardware. On an AI-capable laptop with a Qualcomm or Intel NPU, Qwen3:8b generates a typical worksheet in approximately [INSERT TIME WHEN MEASURED]. On standard hardware without an NPU, generation falls back to CPU inference which is significantly slower. Teachers should be made aware that generation time depends on their hardware.

Compatibility

The dual delivery method — Android app over Wi-Fi and email-based PDF for non-Android devices — ensures the system is not restricted to a single hardware ecosystem. The email workflow was validated with two Richard Cloudesley School teachers using Kindle devices and independently confirmed by Atia Rafiq using a reMarkable tablet. This makes Manuscripta deployable in schools with mixed E-ink device inventories without requiring hardware standardisation.

Maintainability

The Android application follows Clean Architecture with strict separation of data, domain and presentation layers, enforced by the CI pipeline which requires 95% unit test coverage on every pull request. The 126-test, 33,532-line combined test suite provides a strong regression safety net. The Windows application uses a centralised RuntimeDependencyManager for AI dependencies, meaning new models or tools can be added without duplicating management logic.

Project Management

The team used GitHub Issues and pull requests for task tracking, with conventional commit messages and path-based CI/CD separating Android and Windows workflows. The monorepo structure with a shared API contract in api-spec/ enabled parallel development between the two sub-teams without blocking dependencies.

The development cutoff was met on 13 March 2026. Areas where project management could have been stronger include the absence of formal branch protection rules — pull request reviews were voluntary rather than enforced — and the lack of performance benchmarking infrastructure, which was identified late in the project timeline.

Future Work

Priority 1 — Student user testing

The most significant gap in the project is the absence of formal testing with SEN students. Partnering with Richard Cloudesley School to conduct a structured classroom trial would provide direct evidence of the system's effectiveness for the target user group and identify usability issues that teacher testing cannot surface.

Priority 2 — Performance validation at scale

Load testing with 30 simultaneous devices on a standard school Wi-Fi network would validate the non-functional requirements that were specified but not measured. This would also identify any bottlenecks in the TCP distribution pipeline under realistic classroom conditions.

Priority 3 — AI output quality framework

A systematic framework for evaluating AI output quality — including factual accuracy checks, reading level validation and consistency measurement across repeated prompts — would provide the evidence base needed to deploy the system with confidence in a real classroom. The TA recommendation to document example prompts and their outputs in an appendix is noted as a starting point.

Priority 4 — Complete should-have features

Text-to-speech, export of session data, vocabulary highlighting and differentiated deployment to student groups were identified as should-have requirements and partially implemented. Completing these would meaningfully improve the system's accessibility and teacher utility.

Priority 5 — Handwriting recognition

Identified during the second stakeholder meeting with Dean Mohamedally and validated by Atia Rafiq's professional workflow, handwriting recognition for returned PDFs remains the most technically ambitious undelivered requirement. The ONNX model file exists in the codebase but is not yet integrated. This would unlock fully automated marking of non-Android device submissions.

Priority 6 — Microsoft Store packaging

The Windows application was specified to be distributed via the Microsoft Store. The Squirrel installer infrastructure is in place. Completing the Store submission process would make the system accessible to schools without requiring manual installation from a developer build.
