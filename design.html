UI Design

User interface design principles and iterative design process for Manuscripta's dual-app platform.

Design Principles

Design principles guide interface creation to ensure clarity, consistency, and minimal cognitive load for users. For Manuscripta's dual-app system serving neurodiverse students and teachers, we prioritized the following:

Principle Description
Simplicity Interfaces minimized visual clutter and distractions. For students with autism, the E-ink display shows only essential elements: content and navigation. Teacher interface focuses on core tasks: content generation, distribution, and monitoring.
Consistency Visual elements maintain predictable behavior across both applications. Consistent typography, button styles, and layout patterns reduce learning curves and prevent confusion.
Accessibility High contrast ratios (7:1 minimum) for E-ink displays, adjustable text sizes, and optional text-to-speech ensure students with varying sensory and cognitive needs can interact comfortably. Reading level adjustments support comprehension across ability levels.
Feedback Real-time feedback informs users of system status. Students see immediate handwriting recognition responses. Teachers receive live progress updates and help request notifications. Loading states and confirmation messages maintain transparency.
Error Tolerance System handles mistakes gracefully. Auto-save prevents work loss during network drops. Teachers can regenerate content if initial AI output doesn't meet needs. Students can request help without penalty.

Sketches

Student Interface Evolution

Our initial student interface sketches focused on accessibility requirements identified through persona research: alternative input methods for motor disabilities, adaptable cognitive levels for mixed-ability learners, high contrast for reading difficulties, and self-paced progress through tasks.

After stakeholder consultations, we identified usability issues and refined our approach.

Excessive on-screen buttons cause overstimulation for autistic students. 87% show signs of over-stimulation when using tech. Simplified interface by removing accessibility controls from student view. Moved settings to teacher interface for pre-configuration. 2
Many autistic students struggle with reading comprehension. Tailored assistance needed. Added reading level adjustment features for both teacher (class-wide) and student (individual) interfaces with "Simplify" button. 3

Minimal Interface Elements: Removed accessibility controls and reading level adjustments from student view to reduce clutter. Teachers configure these settings in advance based on student needs.

Reading Level Controls: Added dual-level reading adjustment. Teachers set default class reading level, while students can individually simplify text on their devices for personalized support.

Teacher Interface Evolution

Our initial teacher interface enabled AI content generation, lesson deployment, and progress monitoring. From personas, we recognized teachers may lack AI tool familiarity, so we prioritized straightforward navigation.

The interface included student dashboard, lesson creation fields, and analytics for performance visualization. Stakeholder feedback revealed areas for improvement.

Improved Sketch

Issue Uncovered Changes to Our Sketch Severity
Layout was understandable but lacked clear guidance for new users. Renamed tabs descriptively and added structured onboarding prompts for first-time users. 1
Large "lesson plan" text box was ambiguous. Teachers unsure what to input. Broke lesson input into smaller labeled fields (title, subject, content, preview) to guide creation process. 3

Structured Lesson Input Fields: Replaced large text box with labeled fields guiding teachers through content creation step-by-step.

Clear Navigation and Onboarding: Renamed tabs descriptively and added onboarding guidance for first-time users. Added AI assistant for real-time help.

Reading Level Controls: Added class-wide age group slider allowing teachers to set default reading level for all distributed content.

Severity ranges from 1 (lowest) to 3 (highest).

Prototypes

Initial Prototype

We translated sketches into interactive HTML/CSS prototypes simulating both student and teacher interfaces. These prototypes were shared with stakeholders via email for feedback on usability and feature requirements.

Student Prototype

Teacher Prototype

Stakeholder feedback identified three key areas for improvement:

Issue Uncovered Severity
"Ability to lock tablets to get students' attention when needed." 3
"Can we structure content as Units containing Lessons?" 2
"It would be nice if students could easily switch between lesson and quiz." 2

Improved Prototype

We addressed stakeholder feedback through three key improvements:

Classroom Attention Management: Added "Lock All Screens" button enabling teachers to pause all student tablets with one click for whole-class instruction or announcements.

Hierarchical Content Organization: Implemented Unit/Lesson structure allowing teachers to organize content hierarchically. Teachers specify unit and lesson numbers during creation, then deploy via organized dropdowns in classroom control.

Seamless Lesson-Quiz Navigation: Added toggle functionality allowing students to switch between lesson content and quiz questions without returning to main menu, reducing friction and maintaining focus.

Final Evaluation from Educational Sector Contacts

  • Student status should use color-coded blocks rather than small text labels for clearer visual representation
  • Terms like lesson, quiz, and worksheet require clearer definitions
  • Lesson plans and teaching materials should support optional numbering
