System Architecture

Technical architecture, system design, and implementation details for Manuscripta's dual-app platform.

System Overview

Manuscripta is a distributed client-server system consisting of two applications: a Windows teacher application acting as the server, and Android student applications acting as clients. All communication occurs exclusively over a local area network with no cloud dependency, ensuring full GDPR compliance and data privacy for students in SEN school environments.

The architecture prioritises three properties simultaneously: low latency for real-time classroom control, offline capability so student work is never lost during network interruptions, and clean separation of concerns to enable independent testing of each layer. These priorities directly reflect requirements gathered from stakeholder interviews with the National Autistic Society and Richard Cloudesley School.

The Windows application was built by Raphael Li and Nemo Shu. The Android application was built by Priya Bargota and Will Stephen.

Figure 1: System Architecture Overview

Windows Teacher Application

The Windows application consists of two tightly coordinated components: a .NET 10.0 ASP.NET Core backend and a React and Electron frontend.

The frontend is solely responsible for starting, monitoring and terminating the backend as a child process using Node.js child_process.spawn. The backend is published as a self-contained win-x64 executable bundled within the Electron application as an extraResource. This means the teacher installs and runs a single Electron application — the backend starts automatically, with health checks and exponential backoff restart handled transparently. Dynamic port allocation ensures the application starts even if default ports are unavailable.

The backend exposes a SignalR hub at /TeacherPortalHub on port 5910, enabling real-time bidirectional communication between the backend and the frontend. The frontend subscribes to events including UpdateDeviceStatus, HandRaised, DistributionFailed and RefreshResponses, meaning the teacher dashboard updates automatically without any manual refresh. Classroom control actions including PairDevices, DeployMaterial, LockDevices and CreateFeedback are all dispatched through this hub.

AI Integration

Material generation uses a two-model approach running entirely on the teacher's laptop via Ollama. Qwen3:8b is selected as the primary model for material generation because IBM Granite struggled to produce correctly formatted materials, particularly those containing questions. Qwen3 runs in reasonable time on the teacher's Windows laptop and produces well-structured output. IBM Granite 4.0 is retained as a fallback for hardware that cannot support Qwen3, and is used for smaller tasks such as generating feedback on student responses.

For source document processing, ChromaDB is used as a vector database implementing a standard RAG pipeline. Source documents such as textbooks can easily exceed the context window of local language models when passed directly. ChromaDB solves this by chunking documents into embeddings that are stored and retrieved selectively, meaning only the most relevant passages are passed to the model during generation.

Runtime dependencies including Ollama, ChromaDB and rmapi are managed through a centralised RuntimeDependencyManager system. An abstract base class provides a generic detection and installation pattern, with each dependency subclassing from it. This means new dependencies can be added without duplicating management logic.

Content delivery to non-Android E-ink devices is handled through two additional integrations: rmapi for reMarkable tablets and SMTP for Kindle email delivery, supporting the parallel delivery workflow validated with domain expert Atia Rafiq.

Android Student Application

The Android application follows Clean Architecture, dividing the codebase into three strictly separated layers.

The presentation layer contains MainActivity and MainViewModel, using ViewBinding and LiveData to ensure the UI reacts automatically to state changes without directly accessing data sources. The domain layer contains business logic and bidirectional mappers between Room entities and domain models. Entities are immutable with a single constructor, suitable only for Room persistence, while domain models include convenience constructors and input validation that throws exceptions immediately on invalid state. The data layer contains repository implementations, Room DAOs and network DTOs.

Network Layer

The Android application uses three communication channels simultaneously, each chosen for specific reasons:

Channel Port Protocol Rationale
HTTP via Retrofit 5911 REST Material distribution and response submission involve large payloads — REST is reliable and well-suited to text-heavy content
TCP via Java Sockets 5912 Binary Control signals such as screen lock and material distribution require guaranteed delivery and low latency
UDP via DatagramSocket 5913 Broadcast Device discovery uses broadcast so tablets can locate the teacher's laptop without manual IP configuration

HTTP Layer

The HTTP layer uses a chain of three OkHttp interceptors applied in a specific order: authentication first (stamps a device ID header on every request so the server identifies the tablet), logging second (tracks all requests and responses with timing metrics), and error handling last (standardises error responses after all modifications). A RetryInterceptor sits at the front of the chain so every retry passes through all three interceptors, ensuring auth headers are re-applied and errors are logged on each attempt. The retry policy retries on 5xx errors and 408 and 429 responses, but not on other 4xx errors since permanent client errors will not resolve with retries.

TCP Layer

The TCP layer is coordinated by HeartbeatManager, which acts as a single source of truth for all TCP message dispatch. HeartbeatManager owns DISTRIBUTE_MATERIAL and RETURN_FEEDBACK dispatch, sitting between TcpSocketManager and the repositories. This prevents direct coupling between the network layer and the data layer.

The binary TCP message format uses a 1-byte opcode followed by a variable-length operand:

Opcode Name Direction Purpose
0x10 STATUS_UPDATE Client → Server Heartbeat every 3 seconds
0x05 DISTRIBUTE_MATERIAL Server → Client Trigger material fetch
0x12 DISTRIBUTE_ACK Client → Server Confirm receipt per material
0x11 HAND_RAISED Client → Server Help request signal
0x01 LOCK_SCREEN Server → Client Remote screen lock
0x02 UNLOCK_SCREEN Server → Client Remote screen unlock

Acknowledgements are sent per material rather than per batch. This allows the Windows dashboard to display the exact distribution state of each individual material to the teacher, rather than a single batch confirmation.

The heartbeat interval is 3 seconds. The Windows application marks a device as disconnected after 10 seconds of silence. The Android application unpairs after 60 seconds of no acknowledgements.

Design Patterns

Pattern Where Used Rationale
Clean Architecture Android Separation of data, domain and presentation layers enforces testability — domain models can be tested without mocking Room, achieving 96% instruction coverage
Repository Pattern Android and Windows Abstracts all data access behind interfaces, providing a single source of truth and enabling mocking for unit tests
MVVM Android UI Decouples ViewModels from Activities, ensuring UI logic can be tested independently of Android framework classes
Observer Pattern LiveData (Android), SignalR (Windows) Real-time UI updates without polling — frontend subscribes to backend events and reacts automatically
Strategy Pattern SyncEngine (DefaultSyncEngine, NetworkSyncEngine) Swappable sync implementations allow offline and online sync behaviour to be substituted without changing calling code
Process Lifecycle Management Electron frontend managing .NET backend Single executable deployment — frontend spawns, monitors and restarts backend transparently with exponential backoff
Dependency Injection Hilt on Android, built-in DI on .NET Loose coupling between components enables independent testing and substitution of implementations
Chain of Responsibility OkHttp interceptor chain Auth, logging and error handling applied in guaranteed order with clear separation of each concern

API Contract

The HTTP REST API runs on port 5911. Key endpoints:

Method Endpoint Purpose
POST /pair Device registration and pairing
GET /distribution/{deviceId} Fetch assigned materials
POST /responses Submit student answer
GET /config/{deviceId} Fetch tablet configuration
GET /feedback/{deviceId} Retrieve feedback on responses

All device and entity IDs are validated as GUIDs server-side. Invalid IDs return 404 Not Found with a generic message rather than 400 Bad Request, avoiding information leakage about ID validity to unauthenticated callers.
